## 审计方法论

### 设计原则

| 维度 | 选择 | 理由 |
|------|------|------|
| Agent 数 | 4 个并行 explore agent | 4 个区域正交，无相互依赖；避免单个 agent 跨区域认知负担 |
| 任务粒度 | 每个 agent 限 1 区域 | 输入 prompt ≤ 30 行，输出 ≤ 50 行偏差表 |
| 严重度映射 | 🔴 P0 / 🟠 P1 / 🟡 P2 / 🟢 P3 | 与 SSOT §3 修复建议 P0/P1/P2/P3 完全对齐 |
| 表格格式 | 6 列（偏差 / SSOT 描述 / 实际状态 / 严重度 / 证据 / 建议）| 表格形式便于跨章节对比，每行带证据便于复核 |
| 沉淀路径 | `docs/02_architecture/audit-reports/` | 集中存放；与 SSOT 同目录无需跨目录跳转；未来 v0.1.7+ 报告追加为 `v0.1.7-audit.md` 等 |

### Agent prompt 模板

每个 agent prompt 含 6 段：
1. **CONTEXT**: 项目背景 + 审计目标章节
2. **GOAL**: 具体要核对的 SSOT 描述项
3. **DOWNSTREAM**: 报告将用于 v0.1.6 SSOT 勘误
4. **REQUEST**: 列出具体要 grep/read/ls 的文件 + 输出格式
5. **SCOPE**: 明确边界（不修代码，不修 SSOT，只读）
6. **SKIP**: 明确不审计的相邻区域

### 与 v0.1.2 勘误对比

| 维度 | v0.1.2 (commit `4e5d5ea`) | v0.1.6 (本次) |
|------|---------------------------|----------------|
| Agent 数 | 4 (并行) | 4 (并行) |
| 覆盖区域 | §1.3-1.6 + 附录 A IOCTL 编号 | §1.2 仿真层 + §1.7 测试 + §1.8 空白 + 附录 A struct 字段 |
| 偏差数 | 10 项 | 25 项 |
| 最高严重度 | P0 勘误（见 commit msg）| **1 P0**（A4 #1, gpu_pushbuffer_args 缺 va_space_handle）|
| 报告沉淀 | 无独立报告（仅 commit msg）| `audit-reports/v0.1.6-audit.md` |
